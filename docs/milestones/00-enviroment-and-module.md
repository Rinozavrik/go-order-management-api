# Milestone 00 — Учебная среда и Go-модуль

## Цель

Подготовить минимальный воспроизводимый репозиторий для обучения Go и разобраться с базовой моделью инструментов языка.

После завершения milestone должно быть понятно:

- как Go определяет модуль;
- чем модуль отличается от package;
- как запускаются сборка и тесты;
- где хранятся инструкции для Codex;
- как Obsidian, GitHub, Codex и ChatGPT взаимодействуют через репозиторий.

## Ментальная модель

Репозиторий является единым источником истины для кода и учебной документации.

Go-проект состоит из одного или нескольких packages, объединённых в module.

`go.mod` описывает:

- путь модуля;
- используемую версию Go;
- прямые и косвенные зависимости.

Codex читает содержимое репозитория и инструкции из `AGENTS.md`.

Obsidian используется как интерфейс для просмотра и связывания Markdown-документации.

## Сравнение с Node.js

Упрощённая аналогия:

| Node.js                | Go                                     |
| ---------------------- | -------------------------------------- |
| `package.json`         | `go.mod`                               |
| npm package            | Go module                              |
| папка с файлами модуля | package                                |
| `npm test`             | `go test ./...`                        |
| ESLint                 | частично `go vet` и другие анализаторы |
| Prettier               | `gofmt`                                |
| `node_modules`         | module cache Go                        |

Аналогия неполная.

В частности:

- Go module и npm package похожи по назначению, но модель импорта и зависимостей отличается;
- package в Go является языковой единицей организации кода, а не просто директорией;
- `gofmt` — стандартный и практически обязательный форматтер, а не вопрос вкуса команды.

## Теория

Нужно разобраться со следующими вопросами:

- что входит в Go toolchain;
- что такое Go module;
- какую роль играет `go.mod`;
- чем module отличается от package;
- как определяется import path;
- зачем нужен package `main`;
- что делает функция `main`;
- зачем существует директория `internal`;
- как работают команды:
    - `go mod init`;
    - `go mod tidy`;
    - `go test`;
    - `go vet`;
    - `gofmt`;

- почему Go-проекты больше не обязаны находиться внутри `GOPATH`.

## Scope

В рамках milestone необходимо:

- установить Go;
- проверить версию Go;
- клонировать GitHub-репозиторий;
- инициализировать Go module;
- добавить `AGENTS.md`;
- создать минимальную структуру документации;
- открыть корень репозитория как Obsidian vault;
- открыть репозиторий в Codex;
- проверить, что Codex понимает инструкции;
- добавить минимальный исполняемый package;
- добавить один небольшой внутренний package;
- написать первый table-driven test;
- настроить базовые автоматические проверки;
- сделать первый commit и push.

## Не входит в scope

На этом этапе не добавляются:

- HTTP-сервер;
- PostgreSQL;
- MongoDB;
- Redis;
- RabbitMQ;
- NATS;
- Docker;
- Kubernetes;
- конфигурационные библиотеки;
- dependency injection container;
- generic repository;
- service layer;
- сложная clean architecture;
- goroutines;
- channels;
- middleware;
- observability.

Причина проста: инфраструктура должна появляться только тогда, когда она нужна текущей задаче.

```text
go-backend-learning/

│   └── workflows/
├── cmd/

├── docs/
│   ├── concepts/
│   ├── decisions/

│   └── questions.md
├── internal/
├── AGENTS.md
├── README.md
├── go.mod
└── .gitignore
```

`internal/` пока не нужно наполнять заранее сложной структурой.

Packages должны появляться только под конкретную ответственность.

## Практическая задача

Подготовить репозиторий так, чтобы он содержал:

- один исполняемый package в `cmd/app`;
- один небольшой package внутри `internal`;
- минимум одну функцию;
- table-driven test для этой функции;
- инструкции для Codex;
- учебную документацию;
- базовый workflow автоматических проверок.

Полное решение задачи заранее не реализовывать.

## Ограничения

- не использовать сторонние библиотеки;
- не создавать интерфейсы;
- не использовать глобальное изменяемое состояние;
- не создавать `controllers`, `services` и `repositories`;
- не добавлять DI-контейнер;
- не добавлять Docker;
- не использовать `panic` для обычной обработки ошибок;
- весь Go-код должен быть отформатирован через `gofmt`.

## Критерии готовности

Milestone считается выполненным, если:

- `go version` показывает установленную версию Go;
- `go.mod` содержит корректный module path;
- репозиторий открывается в Codex;
- Codex корректно пересказывает правила из `AGENTS.md`;
- репозиторий открывается как Obsidian vault;
- `docs/index.md` является стартовой учебной страницей;
- существует один исполняемый package;
- существует один внутренний package;
- написан минимум один table-driven test;
- все тесты проходят;
- `go vet ./...` не сообщает ошибок;
- после `gofmt` нет незакоммиченных форматирующих изменений;
- все файлы добавлены в Git;
- commit отправлен в GitHub.

## Команды проверки

```bash
go mod tidy
gofmt -w .
go test ./...
go vet ./...
git status
```

На этом этапе `go test -race` ещё не является обязательным, поскольку конкурентного кода нет.

## Definition of Done

Репозиторий можно клонировать на другую машину и проверить стандартными командами без скрытого локального состояния.

Новый разработчик или Codex должны понять:

- назначение проекта;
- текущий milestone;
- ограничения текущего этапа;
- доступные команды проверки;
- что нельзя добавлять раньше времени.
