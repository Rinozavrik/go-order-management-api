# Процесс обучения Go

## Назначение

Этот документ описывает рабочий процесс обучения Go с использованием ChatGPT, Codex, Obsidian и GitHub.

Цель — не автоматизировать обучение, а распределить роли инструментов так, чтобы я самостоятельно писал код, понимал решения и сохранял полученные знания.

## Источник истины

Учебный репозиторий является единым источником истины.

В нём хранятся:

- исходный код;
- текущий milestone;
- учебные заметки;
- результаты code review;
- архитектурные решения;
- открытые вопросы;
- история изменений.

Чаты ChatGPT и Codex используются как рабочие сессии. Важные результаты из чатов должны переноситься в репозиторий.

## Роли

### Я

Я отвечаю за:

- самостоятельную первую реализацию;
- принятие и объяснение решений;
- исправление найденных ошибок;
- подтверждение завершения темы;
- оценку собственного понимания.

### ChatGPT

ChatGPT используется для:

- объяснения тем;
- сравнения Go с Node.js и TypeScript;
- постановки практических задач;
- определения scope и критериев готовности;
- разбора замечаний Codex;
- формирования 3–5 карточек на понимание темы.

ChatGPT не выдаёт полное решение практической задачи до моей попытки, если я прямо не попросил об этом.

### Codex

Codex используется для:

- чтения репозитория;
- запуска проверок;
- code review;
- поиска ошибок и неидиоматичного Go;
- внесения конкретных ограниченных изменений;
- добавления согласованных карточек и обновления документов.

Codex работает по правилам из `../AGENTS.md` и не пишет первую реализацию вместо меня.

Готовые команды находятся в [[codex-prompts]].

### Obsidian

Obsidian используется как интерфейс для учебной документации:

- навигация по темам;
- связанные заметки;
- карточки Spaced Repetition;
- шаблоны Templater;
- схемы Excalidraw.

Vault открывается из директории `docs`.

### GitHub

GitHub используется для:

- хранения кода и документации;
- истории изменений;
- резервного копирования;
- CI;
- доступа ChatGPT и Codex к актуальному состоянию проекта.

Зелёный CI подтверждает технические проверки, но не доказывает понимание темы.

## Рабочий цикл одной темы

1. ChatGPT объясняет тему и формирует практическую задачу.
2. Я создаю или дополняю заметку в `concepts`.
3. Я самостоятельно пишу код.
4. Codex проверяет код по `AGENTS.md` и текущему milestone.
5. ChatGPT помогает разобрать замечания Codex.
6. Я исправляю код и повторно запускаю проверки.
7. ChatGPT формирует 3–5 карточек на понимание темы.
8. Codex добавляет карточки в конец concept note.
9. Obsidian Spaced Repetition используется для повторения.
10. В `reviews` фиксируются реальные ошибки и выводы.
11. После завершения этапа обновляется `index.md`.
12. Изменения фиксируются отдельным логическим commit.

### Правила

- Codex не пишет первую реализацию вместо меня.
- Карточки проверяют понимание, а не знание синтаксиса.
- `index.md` обновляется только при изменении milestone или текущего статуса.
- Не каждое замечание требует отдельной заметки.
- Не каждое решение требует ADR.
- Документация не должна занимать больше времени, чем практика.

## Обновление документов

### `index.md`

Обновляется, когда:

- начат или завершён milestone;
- изменилась текущая задача;
- появился значимый review;
- изменился общий статус проекта.

### `concepts/`

Обновляется после:

- изучения темы;
- практического применения;
- важного уточнения;
- добавления карточек.

Заметка должна отражать моё понимание, а не копировать ответ ChatGPT целиком.

### `reviews/`

Создаётся после значимого code review.

В заметку попадают:

- реальные проблемы;
- почему они важны;
- как они исправлены;
- какие выводы нужно запомнить.

Полный ответ Codex копировать не нужно.

### `questions.md`

Используется как inbox. После разбора вопрос закрывается или превращается в concept note.

### `decisions/`

ADR создаётся только при значимом выборе между альтернативами с заметными последствиями.

## Базовые проверки

```bash
gofmt -w .
go test ./...
go vet ./...
```

Для конкурентного кода:

```bash
go test -race ./...
```

## Завершение milestone

Milestone считается завершённым, когда выполнены три группы условий.

### Технические

- код компилируется;
- тесты проходят;
- `go vet` проходит;
- код отформатирован;
- CI зелёный;
- выполнены проверки текущего milestone.

### Архитектурные

- нет необоснованных абстракций;
- package boundaries понятны;
- ошибки обработаны;
- ресурсы корректно закрываются;
- конкурентный код не содержит известных гонок и утечек goroutine.

### Учебные

Я могу объяснить:

- что реализовано;
- как это работает;
- почему выбран такой подход;
- где я ошибся;
- чем решение отличается от привычного Node.js-подхода;
- какие production-риски и компромиссы существуют.

Зелёные тесты без понимания темы не завершают milestone.

## Карточки для повторения

После завершения темы ChatGPT формирует 3–5 карточек.

Карточки должны проверять:

- ментальную модель;
- причинно-следственные связи;
- trade-offs;
- failure modes;
- production-поведение.

Формат:

```text
Вопрос::Краткий ответ
```

Codex добавляет карточки только по явной команде и не создаёт дубликаты.
