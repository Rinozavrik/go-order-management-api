# Процесс обучения Go

## Назначение документа

Этот документ описывает, как организовано обучение Go с использованием:

- ChatGPT;
- Codex;
- Obsidian;
- GitHub;
- учебного репозитория.

Цель системы — не автоматизировать обучение целиком, а разделить роли инструментов так, чтобы:

- теория не смешивалась с реализацией;
- код не существовал отдельно от объяснений;
- ошибки сохранялись и повторно анализировались;
- текущий прогресс был виден;
- Codex не выполнял учебные задачи вместо обучающегося;
- репозиторий оставался единым источником истины.

---

## Основной принцип

Учебный репозиторий является центральной точкой всей системы.

В нём хранятся:

- исходный код;
- текущий milestone;
- учебная документация;
- архитектурные решения;
- результаты code review;
- список открытых вопросов;
- инструкции для Codex;
- история изменений.

Чаты ChatGPT и Codex используются как рабочие сессии, но не считаются долговременным источником истины.

---

## Компоненты системы

### ChatGPT

ChatGPT используется как преподаватель, технический консультант и архитектурный ревьюер.

Основные задачи ChatGPT:

- объяснять новые темы;
- формировать ментальную модель;
- сравнивать Go с Node.js и TypeScript;
- указывать, где привычные Node.js-подходы неидиоматичны для Go;
- определять scope milestone;
- формулировать практические задачи;
- задавать ограничения и критерии готовности;
- разбирать спорные архитектурные решения;
- объяснять ошибки после code review;
- формировать вопросы для повторения;
- помогать обновлять учебную документацию.

ChatGPT не должен:

- сразу выдавать полное решение учебной задачи;
- скрывать важные детали за фреймворками;
- механически переносить NestJS-паттерны в Go;
- добавлять архитектурные слои без практической необходимости;
- считать тему освоенной только потому, что код компилируется.

---

### Codex

Codex используется как агент, работающий непосредственно с репозиторием.

Основные задачи Codex:

- читать код и документацию;
- учитывать правила из `AGENTS.md`;
- анализировать текущий milestone;
- запускать проверки;
- проводить code review;
- находить ошибки;
- проверять идиоматичность Go;
- проверять обработку ошибок;
- проверять `context.Context`;
- проверять управление ресурсами;
- проверять конкурентный доступ;
- проверять жизненный цикл goroutine;
- проверять package boundaries;
- проверять тесты;
- вносить ограниченные изменения по явной команде;
- обновлять документацию по конкретной задаче.

Codex не должен:

- выполнять практическую задачу целиком до самостоятельной попытки;
- добавлять библиотеки без причины;
- создавать интерфейсы заранее;
- создавать generic repository;
- добавлять DI-контейнер;
- копировать структуру `controllers/services/repositories`;
- отмечать учебные пункты выполненными без подтверждения;
- придумывать прогресс, которого нет.

---

### Obsidian

Obsidian используется как интерфейс для учебной базы знаний.

Основные задачи Obsidian:

- отображать Markdown-документацию;
- связывать темы между собой;
- показывать текущий milestone;
- хранить заметки по концепциям;
- хранить вопросы;
- хранить выводы после ревью;
- визуализировать архитектурные связи;
- помогать повторять материал.

Obsidian не является отдельным источником истины.

Vault открывается из корня учебного репозитория или из директории `docs`.

Все значимые заметки должны храниться в Git.

---

### GitHub

GitHub используется для:

- хранения репозитория;
- истории изменений;
- резервного копирования;
- code review;
- автоматических проверок;
- синхронизации между локальной машиной, Codex и ChatGPT;
- фиксации завершённых milestones.

GitHub не определяет учебный прогресс автоматически.

Зелёный CI означает, что технические проверки прошли, но не означает, что тема полностью понята.

---

## Источник истины

Приоритет источников:

1. Репозиторий.
2. Текущий milestone.
3. `AGENTS.md`.
4. Учебная документация.
5. Последний code review.
6. Текущий чат.

Если информация в чате противоречит актуальному состоянию репозитория, приоритет имеет репозиторий.

Если milestone изменился, это должно быть зафиксировано в документации.

---

## Структура учебной документации

```text
docs/
├── index.md
├── learning-process.md
├── questions.md
├── concepts/
├── decisions/
├── milestones/
├── reviews/
└── daily/
```

### `docs/index.md`

Главная учебная панель.

Содержит:

- текущий milestone;
- текущую задачу;
- общий прогресс;
- ссылки на основные разделы;
- ссылки на последнее review;
- ближайший следующий этап.

Не должен содержать подробные объяснения и длинную историю изменений.

---

### `docs/learning-process.md`

Описывает организацию обучения.

Содержит:

- роли инструментов;
- правила взаимодействия;
- учебный цикл;
- правила обновления документации;
- правила работы с Codex.

Меняется редко.

---

### `docs/questions.md`

Содержит открытые вопросы.

Используется как inbox.

Вопросы могут быть:

- техническими;
- архитектурными;
- связанными с runtime;
- связанными с инструментами Go;
- связанными с отличиями от Node.js.

После разбора вопрос либо переносится в `Resolved`, либо превращается в отдельную заметку в `docs/concepts`.

---

### `docs/concepts`

Содержит заметки по отдельным темам.

Примеры:

- modules;
- packages;
- visibility;
- pointers;
- errors;
- interfaces;
- context;
- goroutines;
- channels;
- testing;
- profiling.

Каждая заметка должна по возможности содержать:

- ментальную модель;
- сравнение с Node.js;
- минимальный пример;
- неидиоматичные подходы;
- production-нюансы;
- связанные темы;
- вопросы для повторения.

---

### `docs/milestones`

Содержит учебные этапы.

Каждый milestone должен включать:

- цель;
- теорию;
- scope;
- out of scope;
- практическую задачу;
- ограничения;
- критерии готовности;
- definition of done;
- команды проверки;
- вопросы для собеседования.

---

### `docs/reviews`

Содержит результаты code review.

Каждое review должно включать:

- что проверялось;
- найденные проблемы;
- серьёзность проблем;
- объяснение причин;
- исправления;
- неидиоматичные конструкции;
- выводы;
- повторяющиеся ошибки.

---

### `docs/decisions`

Содержит архитектурные решения в формате ADR.

Каждый ADR должен включать:

- контекст;
- решение;
- альтернативы;
- последствия;
- условия пересмотра.

---

### `docs/daily`

Используется опционально.

Короткая ежедневная запись может содержать:

- что изучено;
- что реализовано;
- что было непонятно;
- ошибка дня;
- следующий шаг.

Ежедневные записи не должны превращаться в обязательную бюрократию.

---

## Учебный цикл

Каждая новая тема проходит через одинаковый процесс.

### Этап 1. Объяснение темы

ChatGPT объясняет:

- ментальную модель;
- терминологию;
- отличия от Node.js и TypeScript;
- типичные ошибки;
- production-нюансы.

На этом этапе код может быть минимальным и демонстрационным.

---

### Этап 2. Формирование задачи

Для практической задачи определяются:

- цель;
- scope;
- ограничения;
- критерии готовности;
- запрещённые преждевременные решения.

Полное решение заранее не выдаётся.

---

### Этап 3. Самостоятельная реализация

Обучающийся самостоятельно:

- создаёт файлы;
- пишет код;
- пишет тесты;
- запускает проверки;
- фиксирует вопросы.

Codex не реализует задачу за него.

---

### Этап 4. Проверка Codex

Codex:

- читает `AGENTS.md`;
- читает milestone;
- анализирует изменения;
- запускает проверки;
- делает review;
- классифицирует замечания.

Рекомендуемая классификация:

- blocking;
- important;
- improvement;
- question.

---

### Этап 5. Разбор review

ChatGPT помогает:

- проверить корректность замечаний;
- объяснить причины;
- отделить реальные проблемы от вкусовщины;
- показать идиоматичный подход;
- сравнить с привычками из Node.js;
- определить, что исправлять первым.

---

### Этап 6. Исправления

Обучающийся самостоятельно исправляет основные проблемы.

Codex может внести только конкретные ограниченные изменения по явной команде.

Например:

- исправить один найденный дефект;
- добавить недостающий тест;
- обновить один документ;
- применить конкретное замечание review.

---

### Этап 7. Повторная проверка

После исправлений запускаются:

```bash
gofmt -w .
go test ./...
go vet ./...
```

Для конкурентного кода:

```bash
go test -race ./...
```

При необходимости дополнительно используются:

- benchmarks;
- `pprof`;
- integration tests;
- linters;
- Docker-based checks.

---

### Этап 8. Фиксация результата

После завершения задачи обновляются:

- `docs/reviews`;
- `docs/questions.md`;
- `docs/index.md`;
- текущий milestone;
- при необходимости `docs/concepts`;
- при необходимости ADR.

---

### Этап 9. Commit

Изменения фиксируются в Git.

Commit должен отражать завершённый логический шаг.

Примеры:

```text
chore: initialize Go learning workspace
feat: add first internal package
test: add table-driven tests for parser
docs: complete milestone 00
refactor: simplify package API
```

---

## Правила работы с Codex

### Перед началом задачи

Codex можно попросить:

```text
Read AGENTS.md and the current milestone.

Do not modify any files.

Explain:
- whether the repository is ready;
- which concepts are required;
- which files may need to be created;
- which acceptance criteria will be checked.
```

---

### После самостоятельной реализации

Codex можно попросить:

```text
Review my uncommitted changes against AGENTS.md and the current milestone.

Do not modify files.

Run:
- go test ./...
- go vet ./...
- gofmt checks.

Classify findings as:
- blocking;
- important;
- improvement;
- question.
```

---

### После review

Не следует сразу просить:

```text
Fix everything.
```

Предпочтительный подход:

```text
Apply only findings 1 and 3.

Do not modify unrelated files.

Run the required checks after editing.
```

---

## Правила обновления `docs/index.md`

`docs/index.md` обновляется только когда:

- начат новый milestone;
- завершён milestone;
- изменилась текущая задача;
- появился новый важный review;
- изменился общий статус репозитория;
- появились новые основные разделы.

Не нужно обновлять `docs/index.md` после каждого небольшого изменения.

Codex не должен отмечать пункт выполненным без явного подтверждения.

---

## Правила завершения milestone

Milestone считается завершённым только если выполнены три группы условий.

### Технические условия

- код компилируется;
- тесты проходят;
- `go vet` не сообщает ошибок;
- код отформатирован;
- CI проходит.

### Архитектурные условия

- нет необоснованных абстракций;
- package boundaries понятны;
- зависимости явные;
- ошибки обработаны;
- ресурсы закрываются;
- конкурентный код не содержит очевидных утечек и гонок.

### Учебные условия

Обучающийся может объяснить:

- что было реализовано;
- почему выбран такой подход;
- какие альтернативы рассматривались;
- какие ошибки были найдены;
- чем решение отличается от типичного Node.js-подхода;
- какие production-риски существуют.

Зелёные тесты без понимания темы не завершают milestone.

---

## Язык проекта

Используется следующее соглашение:

- исходный код — английский;
- имена packages — английский;
- имена файлов — английский;
- commit messages — английский;
- `AGENTS.md` — английский;
- учебная документация — русский;
- технические термины Go сохраняются на английском.

Примеры терминов:

- goroutine;
- channel;
- error wrapping;
- escape analysis;
- graceful shutdown;
- race condition;
- package boundary.

---

## Принципы архитектуры

На ранних этапах запрещено усложнять проект без причины.

Не добавляются заранее:

- generic repository;
- DI container;
- service layer;
- controller layer;
- event bus abstraction;
- framework-specific architecture;
- лишние interfaces;
- микросервисы.

Архитектура должна расти из требований текущего milestone.

Сначала решается реальная задача, затем появляется абстракция.

---

## Роли человека и инструментов

### Обучающийся отвечает за

- написание первой реализации;
- принятие архитектурных решений;
- объяснение собственного кода;
- подтверждение завершения темы;
- разбор ошибок;
- фиксацию выводов.

### ChatGPT отвечает за

- объяснение;
- постановку задач;
- обучение;
- разбор ошибок;
- архитектурную критику;
- помощь в структурировании знаний.

### Codex отвечает за

- анализ репозитория;
- технические проверки;
- code review;
- ограниченные изменения;
- работу с файлами.

### Obsidian отвечает за

- навигацию;
- визуализацию;
- связывание знаний;
- удобное чтение документации.

### GitHub отвечает за

- хранение;
- историю;
- синхронизацию;
- CI;
- review changes.

---

## Антипаттерны учебного процесса

Следует избегать следующих ситуаций:

- Codex пишет весь milestone;
- ChatGPT сразу выдаёт полное решение;
- документация не обновляется;
- код и заметки хранятся отдельно;
- `index.md` превращается в длинный конспект;
- каждый вопрос становится отдельным файлом;
- Obsidian настраивается дольше, чем изучается Go;
- зелёные тесты принимаются за доказательство понимания;
- архитектура строится по аналогии с NestJS;
- интерфейсы создаются ради моков;
- milestone разрастается без ограничения scope.

---

## Конечная цель процесса

После прохождения программы обучающийся должен уметь самостоятельно:

- спроектировать backend-сервис на Go;
- определить package boundaries;
- выбрать минимально достаточную архитектуру;
- работать с `context.Context`;
- корректно обрабатывать ошибки;
- управлять ресурсами;
- писать конкурентный код;
- находить race conditions;
- писать HTTP API;
- работать с PostgreSQL, MongoDB, Redis и брокерами сообщений;
- писать unit и integration tests;
- профилировать приложение;
- настраивать observability;
- контейнеризировать сервис;
- разворачивать его в Kubernetes;
- объяснять инженерные решения на собеседовании.
