# Go Order Management API

Учебный backend-проект для освоения идиоматичной разработки на Go.

Проект разрабатывается постепенно, небольшими milestones. Сначала используются стандартная библиотека и минимальная архитектура. Инфраструктура и сторонние библиотеки добавляются только тогда, когда они требуются конкретной задачей.

## Цель

Научиться самостоятельно:

- проектировать backend-сервисы на Go;
- определять границы packages;
- писать идиоматичный Go-код;
- работать с ошибками и `context.Context`;
- управлять ресурсами и конкурентным выполнением;
- писать HTTP API;
- работать с PostgreSQL и MongoDB;
- использовать Redis и брокеры сообщений;
- писать unit и integration tests;
- профилировать приложения;
- добавлять observability;
- контейнеризировать и разворачивать сервис.

Цель проекта — не механически переписать NestJS-приложение на Go, а понять принятые в Go подходы к архитектуре, API и управлению зависимостями.

## Инструменты

В процессе обучения используются:

### ChatGPT

Используется для:

- объяснения теории;
- сравнения Go с Node.js и TypeScript;
- постановки практических задач;
- разбора ошибок;
- обсуждения архитектурных решений.

### Codex

Используется для:

- анализа репозитория;
- запуска проверок;
- code review;
- поиска ошибок;
- внесения ограниченных изменений по явной команде.

- milestones;
- вопросы;
- reviews;
- архитектурные решения.

### GitHub

Используется как единый источник истины для:

- кода;
- документации;
- истории изменений;
- автоматических проверок.

## Базовые команды

Форматирование:

```bash
gofmt -w .
```

Тесты:

```bash
go test ./...
```

Статический анализ:

```bash
go vet ./...
```

Проверка конкурентного кода:

```bash
go test -race ./...
```

Последняя команда станет обязательной после появления goroutines и конкурентного доступа к данным.

## Учебный процесс

Каждая тема проходит через один цикл:

1. Разбор теории в ChatGPT.
2. Формирование задачи и критериев готовности.
3. Самостоятельная реализация.
4. Проверка через Codex.
5. Разбор замечаний.
6. Самостоятельное исправление.
7. Повторная проверка.
8. Фиксация выводов в документации.
9. Commit завершённого логического этапа.
